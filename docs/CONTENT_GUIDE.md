# CONTENT_GUIDE.md
## Как писать уроки для приложения

Этот файл управляет написанием контента. Цель — единый формат, исчерпывающее покрытие и проверяемое качество.

## Принципы

1. **Исчерпывающе.** Не замалчиваем ни одной фичи модуля, включая мелочи и удалённые возможности. Если что-то осознанно опускаем — отмечаем явно.
1a. **Теория — для тех, кто знает язык, но не этот стандарт.** Не объясняем, что такое шаблон, ссылка или RAII — ученик это знает. Теория должна раскрывать нюансы и особенности именно новой фичи: как она работает «под капотом», в каких контекстах применяется, какие у неё edge cases, ограничения, взаимодействие с уже знакомыми механизмами языка. Цель — не «узнал, что такая штука есть», а «разбираюсь в ней досконально».

### Чек-лист нюансов (к принципу 1a)

Перед тем как считать теорию исчерпывающей, явно пройти по пунктам ниже —
для каждого либо раскрыть в теории, либо отметить как неприменимый:

- **Unevaluated-контексты.** Как фича ведёт себя в `decltype`, `sizeof`,
  `noexcept`, `requires`? Меняется ли что-то по сравнению с обычным
  (evaluated) использованием?
- **SFINAE / перегрузка / concepts.** Если использование фичи может быть
  некорректным — это hard error всегда, или substitution failure в
  immediate context (а значит, пригодно для SFINAE)?
- **Категория значения.** Если фича производит выражение — это lvalue,
  xvalue или prvalue? Меняется ли это в зависимости от формы записи
  (например, со скобками/без)?
- **Старые альтернативы.** Каким способом эта задача решалась раньше
  (до фичи)? Когда старый способ всё ещё предпочтительнее новой фичи, а
  когда — нет?
- **Ill-formed vs UB vs диагностика.** Что именно происходит при неверном
  использовании — ошибка компиляции, UB в рантайме или диагностика на
  этапе компиляции без UB? C++26 часто переводит старые UB-случаи в
  ill-formed — если это так, явно проговорить.
- **Взаимодействие с уже знакомым.** Как фича сочетается с шаблонами,
  `constexpr`, ссылками/value categories, перегрузкой — то есть с
  механизмами, которые ученик уже знает из C++14/17?

`m1-l1` (Pack indexing) — referenced пример прохода по этому чек-листу: в
него по итогам добавлены разделы про unevaluated/`decltype`
(value category), SFINAE-friendliness out-of-range индекса и сравнение со
старой альтернативой (`std::tuple`/`std::get<N>`).
2. **Каждый пример с проверенным выводом.** Перед тем как зафиксировать ожидаемый вывод, Claude сам компилирует и исполняет код через публичный API Compiler Explorer (скилл `compiling-cpp26-examples`, скрипт `prototype/run-on-godbolt.js`) — без участия человека и без локального тулчейна. Ученик не компилирует — он сверяется с эталоном.
3. **«Предскажи → раскрой».** Лучший формат примера: сначала ученик предсказывает результат, потом видит ответ с разбором. Это и пример, и активная проверка.
4. **Фон только load-bearing.** Базовая планка — C++14 + C++17. Объясняем только то из C++20/23, что нужно для понимания конкретной фичи C++26. Ниже планки — внешняя ссылка, не пишем сами.
5. **Мотивация через ограничение.** Хороший фон/мотивация почти всегда описывает ограничение прошлых стандартов, которое C++26 снимает.
6. **Сверка bleeding-edge.** Точные имена API рефлексии и `std::execution` проверяем по cppreference — синтаксис уточнялся поздно.

## Планка глубины

Объём урока определяется не «размером» фичи как таковой, а числом её
**контекстов использования** — отдельных синтаксических форм, областей
применения, взаимодействий с другими механизмами языка, которые нужно
раскрыть, чтобы покрыть фичу исчерпывающе (принцип 1a). Теория пишется
полностью, без сокращений, независимо от категории — категория влияет
только на число примеров и задач.

| Тип фичи | Контексты | Примеры/задачи |
|----------|-----------|-----------------|
| Мелкая (1–2 контекста использования) | напр. `=delete("reason")`, `_` как имя | 1–2 примера + 2 задачи |
| Средняя (3–4 контекста использования) | напр. pack indexing, контракты, `inplace_vector` | 3–4 примера + 3–4 задачи + challenge |
| Флагман (рефлексия, `std::execution`) | разбит на под-уроки; на каждый — полный шаблон |

`m1-l1` (Pack indexing) — референс для категории «средняя»: 4 контекста
использования (выбор элемента, срез, structured bindings, тип-форма в
сигнатуре) → 4 примера, 3 задачи + challenge, 3 вопроса mastery check.

## Языки контента

Контент хранится отдельно для каждой локали: `content/modules/ru/mN.json` и
`content/modules/en/mN.json`. Схема одинаковая для обоих языков (см. ниже);
переводятся только текстовые поля (`title`, `motivation`, `theory`,
`exercises[].*`, `masteryCheck.questions[].*`, комментарии внутри `code`).
Идентификаторы, `answerIndex`/`answerLine`/`answer`, `expectedOutput`,
`verifiedWith` и сама структура C++-кода должны быть одинаковы между
локалями — это проверяет `prototype/check-i18n-parity.js`.

## Формат данных урока (JSON)

Контент хранится как данные, отдельно от кода. Каждый модуль — один JSON-файл.

```json
{
  "id": "m1",
  "moduleNumber": 1,
  "title": "Эргономика ядра языка",
  "significance": "базовый",
  "prerequisites": [],
  "lessons": [
    {
      "id": "m1-l1",
      "title": "Pack indexing",
      "prerequisites": [],
      "background": null,
      "verifiedWith": { "compilerId": "gsnapshot", "flags": "-std=c++26 -O2" },
      "motivation": "markdown: какую боль решает фича",
      "theory": "markdown: исчерпывающее объяснение",
      "examples": [
        {
          "title": "Базовый",
          "code": "template <typename... T>\nconstexpr auto pick(T... v) { return v...[1]; }",
          "expectedOutput": "20",
          "explanation": "markdown: почему так"
        }
      ],
      "exercises": [
        {
          "type": "predict-output",
          "code": "...",
          "answer": "20",
          "explanation": "..."
        },
        {
          "type": "find-bug",
          "code": "...",
          "answerLine": 2,
          "explanation": "..."
        },
        {
          "type": "choice",
          "prompt": "Что здесь произойдёт?",
          "options": ["...", "...", "..."],
          "answerIndex": 1,
          "explanation": "..."
        }
      ],
      "challenge": {
        "prompt": "markdown: свободное задание",
        "starterCode": "сырой код: полный каркас (#include + main + драйвер) с заглушкой `// твой код` на месте ответа",
        "referenceSolution": "```cpp\n…самодостаточная программа: #include + main, печатает expectedOutput…\n```",
        "expectedOutput": "...",
        "godboltUrl": "https://godbolt.org/..."
      },
      "masteryCheck": {
        "passThreshold": 0.8,
        "questions": [
          { "type": "choice", "prompt": "...", "options": ["..."], "answerIndex": 0 }
        ]
      }
    }
  ]
}
```

Поля `background` и `challenge` опциональны (`null`, если нет). Типы упражнений: `predict-output`, `find-bug`, `choice`. Свободный код — только в `challenge`, не автопроверяется.

`verifiedWith` записывает компилятор и флаги, которыми проверены выводы урока (см. скилл `compiling-cpp26-examples` — в большинстве уроков это `gsnapshot` с `-std=c++26 -O2`, опционально `-fcontracts` для Модуля 4). Тем же компилятором и флагами движок компилирует код ученика в челлендже, чтобы сравнение с `expectedOutput` было корректным (см. `docs/superpowers/specs/2026-06-07-runnable-challenge-design.md`). `challenge.expectedOutput` — это проверенный вывод эталонного решения, полученный тем же прогоном через Compiler Explorer, что и для примеров; без него кнопка «Сверить» не сможет показать вердикт.

## Чек-лист готовности урока (Definition of Done)

- [ ] Есть мотивация (зачем фича).
- [ ] Теория покрывает фичу исчерпывающе.
- [ ] Пройден чек-лист нюансов (принцип 1a): по каждому пункту — либо
      раскрыт в теории, либо явно отмечен как неприменимый к этой фиче.
- [ ] Минимум базовый + продвинутый пример, у каждого проверенный `expectedOutput`.
- [ ] Если в уроке есть `challenge`, у него заполнены `expectedOutput` (проверенный прогоном эталонного решения) и `verifiedWith` на уровне урока — иначе «Сверить» не сможет дать вердикт.
- [ ] `referenceSolution` — это **самодостаточная программа в ```cpp-блоке** (с нужными `#include` и `main`), которая компилируется указанным в `verifiedWith` компилятором и печатает ровно `expectedOutput`. Ученик копирует решение и запускает его, поэтому голого фрагмента без заголовков и `main` недостаточно (он не скомпилируется), а код без ```-ограждения движок отрендерит как искажённый абзац вместо блока кода.
- [ ] У челленджа есть `starterCode` — **сырой** (без ```-ограждения) запускаемый каркас с заглушкой `// твой код` на месте ответа; всё, кроме области ответа (инклюды, `main`, драйвер), побайтово совпадает с `referenceSolution`, чтобы корректно заполненный каркас воспроизводил `expectedOutput`. Проверяется `node prototype/check-starter-scaffold.js`.
- [ ] Минимум одна автопроверяемая задача basic и одна advanced.
- [ ] Фоновая врезка добавлена, если есть зависимость от C++20/23.
- [ ] Есть проверка усвоения.
- [ ] Bleeding-edge API сверены по cppreference.
- [ ] Урок переведён на оба языка (`content/modules/ru/...` и
      `content/modules/en/...`), `node prototype/check-i18n-parity.js`
      проходит без ошибок.

## Рабочий процесс

1. Сначала пишем урок в прозе по `LESSON_TEMPLATE.md`.
2. Claude прогоняет все примеры через Compiler Explorer (скилл `compiling-cpp26-examples`) и фиксирует проверенные выводы.
3. Конвертируем в JSON по схеме выше.
4. Прогоняем по чек-листу DoD.
