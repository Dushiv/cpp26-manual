# Облачная синхронизация прогресса — дизайн

## Контекст и мотивация

Прогресс ученика теперь сохраняется локально (`localStorage`, см. `docs/superpowers/specs/2026-06-08-local-progress-persistence-design.md`) и переживает перезагрузку страницы на одном устройстве. Этого недостаточно для сценария «несколько знакомых людей проходят курс и каждый хочет видеть свой прогресс на разных устройствах» — ради него и проектируется эта фича.

Целевая аудитория — небольшая известная группа людей (не открытая регистрация всех желающих), поэтому решение оптимизируется под простоту и низкие эксплуатационные затраты, а не под масштаб.

**Важно:** это аддитивный слой поверх локального сохранения, а не его замена. Локальное сохранение продолжает работать как раньше — облачная синхронизация добавляется поверх него и является полностью опциональной (вход не обязателен).

## Что добавляется

1. **Вход через OAuth** (Google/GitHub) — через Supabase Auth.
2. **Облачное хранилище** — одна строка в таблице Postgres на ученика, содержащая тот же блок `{ version, data }`, что уже пишется в `localStorage`.
3. **Фоновая синхронизация** — облако становится источником истины после входа; периодически (по таймеру) состояние выгружается в облако, при входе/загрузке — затягивается из облака.

## Архитектура

- **Supabase-проект**: Auth (Google/GitHub OAuth) + таблица `progress`:
  ```sql
  create table progress (
    user_id uuid primary key references auth.users(id),
    blob jsonb not null,
    updated_at timestamptz not null default now()
  );
  -- RLS: пользователь может читать/писать только свою строку
  ```
- **Клиент**: `@supabase/supabase-js`, подключаемый через CDN (esm.sh/unpkg) — без шага сборки, как и весь прототип.
- **Новый модуль `cloudSync`** в `prototype/cpp26-engine.jsx` — обёртка над «затянуть при входе/загрузке» и «выгрузить по таймеру», использующая тот же формат блока, что и `loadProgress`/`saveProgress`.

Локальный слой (`loadProgress`/`saveProgress`, `PROGRESS_KEY`/`PROGRESS_VERSION`) не меняется и продолжает быть единственным механизмом для тех, кто не вошёл в систему.

## Вход / выход и виджет аккаунта

- В шапке приложения (рядом с переключателем строгого режима / ссылкой на зону повторения) появляется **виджет аккаунта**:
  - не вошёл: кнопки «Войти через Google» / «Войти через GitHub»;
  - вошёл: имя/аватар ученика + «Выйти».
- **Вход**: `supabase.auth.signInWithOAuth({ provider })` → редирект к провайдеру и обратно; Supabase сам восстанавливает сессию (хранит её в своём ключе `localStorage`, отдельном от `cpp26-progress`).
- **Первый вход** (для этого `user_id` в таблице `progress` ещё нет строки): текущее **локальное** состояние становится начальным значением облачной строки — тот, кто уже занимался без входа, не теряет прогресс.
- **Повторный вход** (строка уже существует): облачные данные побеждают и перезаписывают локальное состояние и `localStorage` (см. ниже «облако — источник истины»).
- **Выход**: перед очисткой сессии — попытка выгрузить текущий локальный блок (если он отличается от последнего выгруженного), и только потом сессия Supabase очищается. Это нужно, чтобы несинканные изменения не потерялись: при следующем входе (в этом же или другом браузере) «облако побеждает» и затёрло бы их устаревшим состоянием. После очистки сессии приложение продолжает работать на локальных данных как раньше — локальное сохранение никогда не отключается, оно лишь дополняется.
- **Смена ученика на одном устройстве**: вход под другим аккаунтом затягивает *его* облачный блок и перезаписывает локальный — то же правило «облако побеждает», без особых случаев.

## Механизм синхронизации

- **Затягивание (pull)** — при монтировании приложения, если есть активная сессия:
  1. Запросить строку `progress` по `user_id`.
  2. Если строка есть — заменить все пять полей состояния (`cur`, `view`, `exStatus`, `mastery`, `strict`) и `localStorage` облачным блоком (это и есть «облако — источник истины»).
  3. Если строки нет — это первый вход: создать строку из текущего локального блока (см. выше).
- **Выгрузка (push)** — пока ученик в системе, сравнивать текущий блок с последним выгруженным и, если изменился, делать `upsert` в `progress`. Сравнение нужно, чтобы не писать в облако впустую, когда ученик просто читает. Срабатывает по нескольким триггерам:
  - каждые 3 минуты по таймеру (подстраховка);
  - через ~5 секунд после любого изменения прогресса (debounce — чтобы кросс-устройственная синхронизация ощущалась почти мгновенной, а не ждала таймера);
  - при скрытии вкладки (`visibilitychange`) и при закрытии/перезагрузке (`beforeunload`, best-effort, не блокирует UI);
  - при выходе из аккаунта (см. ниже) — этот единственный случай дожидается завершения выгрузки перед `signOut()`.
- Пока `syncOnLogin` не завершился (нет базовой точки сравнения), выгрузка не выполняется — иначе можно отправить в облако локальное состояние «до» затягивания и затереть чужой/более новый облачный блок.
- **Конфликты между устройствами** — простое «последняя запись побеждает» (как и сама модель локального блока — целиком перезаписываемый JSON, без слияния по полям). Если ученик успел поработать офлайн на двух устройствах, верх возьмёт то, что выгрузится позже.

## Обработка ошибок

Сеть недоступна, Supabase не отвечает, конфликт записи — все ошибки перехватываются и проглатываются точно так же, как уже сделано в `loadProgress`/`saveProgress`: приложение продолжает работать на локальном состоянии, синхронизация молча повторится на следующем тике таймера. Никаких сообщений об ошибках ученику — сбой синхронизации для него незаметен, прогресс никогда не теряется (он всегда есть локально).

Офлайн-ученик, вошедший в систему: локальное чтение/запись работает как обычно (локальный слой никогда не обходится), попытки выгрузки в облако молча проваливаются и повторяются при восстановлении соединения.

## Явно вне рамок (YAGNI)

- Слияние прогресса по полям при конфликте — намеренно не делается; «последняя запись побеждает» признано достаточным для небольшой доверенной группы.
- Открытая регистрация / произвольное число учеников — рассчитано на «несколько знакомых людей», а не на масштабирование.
- UI-индикаторы состояния синхронизации («синхронизировано», «ошибка сети» и т.п.) — синхронизация работает молча и не должна отвлекать ученика.
- Кастомный backend — Supabase закрывает и Auth, и хранилище без отдельного серверного кода (RLS вместо ручной авторизации).
