# Локальное сохранение прогресса — дизайн

## Контекст и мотивация

Сейчас прогресс ученика (`cur`, `view`, `exStatus`, `mastery`, `strict` в компоненте `App` прототипа `prototype/cpp26-engine.jsx`) хранится только в памяти React и пропадает при перезагрузке страницы. Это первый шаг из дорожной карты `MASTER_PLAN.md` (M1: модель данных + локальное сохранение прогресса) — фундамент, без которого бессмысленно проектировать облачную синхронизацию между устройствами (это отдельный, более крупный проект, который будет обсуждаться позже).

Цель этой фичи — чтобы прогресс ученика переживал перезагрузку страницы и переоткрытие приложения на том же устройстве, используя `localStorage`.

## Что сохраняется

Все пять полей состояния `App`, формирующих "где ученик находится и что он сделал":

- `cur` — id текущего урока
- `view` — текущий экран (`"lesson"` | список модулей/уроков)
- `exStatus` — статусы решённых упражнений (по id упражнения)
- `mastery` — результаты mastery-проверок (по id урока)
- `strict` — включён ли строгий режим

При следующем открытии приложение должно восстановиться в то же состояние — включая то, на каком уроке/экране остановился ученик.

## Формат хранения

Один JSON-блок под одним ключом `localStorage` — `cpp26-progress`:

```json
{
  "version": 1,
  "data": {
    "cur": "m1-l1",
    "view": "lesson",
    "exStatus": { "m1-l1-ex1": "done" },
    "mastery": { "m1-l1": { "passed": true, "score": 0.9 } },
    "strict": false
  }
}
```

Константа `PROGRESS_VERSION = 1` определяется рядом с форматом.

## Загрузка (при старте `App`)

1. Прочитать `localStorage.getItem("cpp26-progress")`.
2. Распарсить JSON.
3. Если парсинг упал, ключ отсутствует, или `data.version !== PROGRESS_VERSION` — использовать дефолтные значения состояния (как сейчас, "с чистого листа"). Никаких миграций между версиями — несовместимые данные просто отбрасываются.
4. Иначе — использовать значения из `data` как начальные значения для соответствующих `useState`.

## Сохранение

Один `useEffect`, зависящий от всех пяти полей состояния. При изменении любого из них:

1. Собрать объект `{ version: PROGRESS_VERSION, data: { cur, view, exStatus, mastery, strict } }`.
2. Записать его целиком через `localStorage.setItem("cpp26-progress", JSON.stringify(...))`.

Запись синхронная, без debounce: объём данных — единицы килобайт, частота изменений определяется действиями ученика (не кадрами рендеринга), так что перезапись всего блока не создаёт заметной нагрузки.

## Обработка ошибок

И чтение, и запись оборачиваются в `try/catch`. Возможные причины сбоя — приватный режим браузера (доступ к `localStorage` запрещён), переполненная квота хранилища. При любой ошибке приложение продолжает работать без персистентности — то есть ведёт себя так же, как сейчас (прогресс живёт только в памяти и теряется при перезагрузке), без падения или сообщений об ошибке пользователю.

## Версионирование на будущее

При несовместимых изменениях структуры сохраняемых данных (например, смена схемы `exStatus` или `mastery`) достаточно увеличить `PROGRESS_VERSION`. Старые сохранённые данные автоматически отбрасываются при следующей загрузке — миграционная логика не нужна.

## Явно вне рамок (YAGNI)

- Кнопка/возможность сброса прогресса — отложена, прогресс и так не переживает смену браузера/устройства.
- Облачная синхронизация между устройствами и аккаунты (Google/GitHub OAuth) — отдельный проект, который будет спроектирован отдельно после того, как эта локальная модель будет готова и стабильна.
- Миграции между версиями формата — намеренно не реализуются; несовместимые данные просто отбрасываются.
