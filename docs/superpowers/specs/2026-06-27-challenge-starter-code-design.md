# Предзаполненный стартовый код в челлендже — дизайн

**Дата:** 2026-06-27
**Статус:** одобрено к реализации

## Проблема

В челлендже («напиши сам») поле ввода кода стартует **пустым**
([`Challenge`](../../../prototype/cpp26-engine.jsx) — `useState("")`). Кнопки
«Запустить»/«Сверить» компилируют ровно то, что в поле, и «Сверить» сравнивает
вывод с `challenge.expectedOutput`.

`expectedOutput` получен прогоном конкретного эталонного решения с конкретным
`main`/драйвером. Раз поле пустое, ученик должен сам написать все `#include`,
`main` и печать — и почти наверняка напишет драйвер иначе, поэтому вывод
**никогда не совпадёт** с эталоном, а «Сверить» теряет смысл.

## Цель

Класть в редактор готовый **каркас** программы (инклюды + `main` + драйвер,
дающий `expectedOutput`), где не хватает только той части, которую просит
написать/переписать задача. Тогда правильно заполненный каркас по построению
воспроизводит `expectedOutput`, и «Сверить» работает.

## Не-цели (вне рамок)

- Сохранение черновика ученика между визитами (поле остаётся эфемерным).
- Блокировка нередактируемых частей каркаса (textarea редактируется целиком).
- Изменение модели прогресса: челлендж по-прежнему необязателен и не влияет на
  mastery-gating.

## Зафиксированные решения (из брейнсторминга)

1. Поле ввода **предзаполнено и полностью редактируемо** (простой `textarea`).
2. Источник стартового кода — **отдельное поле `starterCode`** на челлендж
   (не выводим из `referenceSolution` маркерами).
3. Метка места для кода ученика — комментарий `// твой код`.
4. Черновик **эфемерный**: при возврате на урок поле снова показывает каркас
   (напечатанное не сохраняется). Челлендж необязателен, прогресс не теряется.
5. Рядом с редактором — ссылка **«Сбросить к началу»**, возвращающая
   `starterCode` (иначе после правок каркас не вернуть).

## Модель данных

В объект `challenge` добавляется строковое поле `starterCode` — **полная
запускаемая программа** (нужные `#include`, `main`, драйвер), в которой только
область ответа заменена на заглушку `// твой код`.

- Хранится как **сырой код, без ```-ограждения** — он попадает в `<textarea>`,
  а не рендерится Markdown'ом (так же, как `examples[].code`).
- Инвариант корректности: `starterCode` — это программа из `referenceSolution`,
  где заменена **только** область ответа; всё остальное (инклюды, `main`,
  драйвер) **побайтово совпадает** с `referenceSolution`. Значит, заполнив
  заглушку эталонным ответом, получаем ровно `referenceSolution`, а он уже
  проверен и печатает `expectedOutput`.
- i18n: между `ru` и `en` `starterCode` структурно идентичен; различаться могут
  только комментарии (как и в остальном коде — см. CONTENT_GUIDE).

### Пример (`m1-l1`, «Pack indexing»)

`referenceSolution` (как есть):
```cpp
#include <print>
#include <cstddef>

template <std::size_t N, typename... Ts>
constexpr auto nth(Ts... ts) { return ts...[N]; }

int main() {
    std::print("{}\n", nth<1>(10, 20, 30));
}
```

`starterCode` (новое поле, сырой код):
```cpp
#include <print>
#include <cstddef>

template <std::size_t N, typename... Ts>
constexpr auto nth(Ts... ts) {
    // твой код
}

int main() {
    std::print("{}\n", nth<1>(10, 20, 30));
}
```

## Изменения в движке

Компонент [`Challenge`](../../../prototype/cpp26-engine.jsx):

1. Сидирование редактора: `const [code, setCode] = useState(ch.starterCode || "")`.
   Фолбэк на `""` — если у челленджа ещё нет `starterCode` (во время выкатки).
2. Кнопка/ссылка **«Сбросить к началу»**: `onClick={() => setCode(ch.starterCode || "")}`.
   Новая строка перевода (`resetToStarter`) в обеих локалях.
3. Поведение «Запустить»/«Сверить» не меняется — компилируют текущее содержимое
   поля. `disabled={!code.trim()}` продолжает работать.
4. Поле остаётся локальным стейтом компонента: при смене урока `Challenge`
   размонтируется и при возврате пересоздаётся с `starterCode` — это и есть
   «эфемерный черновик» из решения 4 (отдельного кода не требует).

Строка-плейсхолдер `solutionPlaceholder` остаётся как фолбэк на случай пустого
`starterCode`.

## Контент

Добавить `starterCode` ко **всем челленджам**: 23 урока × 2 локали = 46 полей.

Правило авторинга на каждый челлендж:

1. Взять `referenceSolution` (сырой код внутри ```-блока).
2. Определить область, которую просит написать/переписать `prompt`.
3. Заменить эту область на `// твой код` (для тела функции) либо на сигнатуру с
   заглушкой — остальное (инклюды, `main`, драйвер) скопировать **дословно**.
4. Записать результат в `starterCode` как сырой код (без ```-ограждения).

Для задач «напиши с нуля» (m4/m5) каркас даёт инклюды, `main` с вызовами и
сигнатуры функций, а тела — заглушки.

## Проверка и парность

1. **Проверка инварианта каркаса (автоматическая).** Скрипт проверяет, что
   каждая значимая (непустая, не-заглушка `// твой код`) строка `starterCode`
   присутствует и среди строк `referenceSolution`. Это гарантирует, что каркас
   (инклюды, `main`, драйвер, сигнатуры) скопирован из проверенного эталона
   дословно: если кто-то поменяет драйвер/инклюд в `starterCode` и забудет в
   `referenceSolution`, строка не найдётся и проверка упадёт. Инвариант
   справедлив независимо от того, где заглушка — в теле функции или внутри
   `main` (для задач «напиши с нуля»).
2. **i18n-парность.** Расширить `prototype/check-i18n-parity.js`, чтобы он
   учитывал `starterCode` (наличие в обеих локалях; структурная идентичность по
   тем же правилам, что и для прочего кода).
3. **e2e.** Тест: при открытии урока поле редактора **не пустое** и содержит
   `starterCode`. Второй тест: вписать эталонный ответ в заглушку и по «Сверить»
   получить вердикт ✓ (вывод совпал с эталоном). Мокать Compiler Explorer не
   обязательно — можно проверить только предзаполнение, а матчинг покрыть на
   одном-двух челленджах с реальным прогоном.

## Объём и выкатка

- Движок + строки перевода + ссылка «Сбросить» — один небольшой заход.
- Контент: 46 полей `starterCode`. Так как `referenceSolution` уже есть и
  проверен, авторинг механический, но требует пер-челлендж-решения, какую
  область заглушать. Можно бить по модулям.
- Проверочный скрипт + e2e + правка парности + CONTENT_GUIDE.
