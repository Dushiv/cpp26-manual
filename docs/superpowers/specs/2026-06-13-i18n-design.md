# i18n design: Russian/English content + UI switcher

## Context

The app is currently Russian-only across two layers:

- **UI chrome** in `prototype/cpp26-engine.jsx` — buttons, statuses, verdicts,
  navigation labels (e.g. "Выполнено", "Запустить", "Сверить", "Верно").
- **Lesson content** in `content/modules/*.json` — motivation, theory,
  examples, exercises, mastery checks, including Russian-language code
  comments inside example code blocks.

This is also open calibration #1 from `docs/PROJECT_OVERVIEW.md`
("язык контента уроков — русский / английский / оба"). This design resolves
it: **both**, with a switcher.

Currently only `m0-l1` and `m1-l1` are written (the rest of modules 1-2 are
stubs); all other modules (3-10) are empty stubs. This is the cheapest point
to lock in the bilingual content shape.

Separately, `prototype/cpp26-engine.jsx` currently inlines `COURSE_DATA` as a
~1000-line JS object that duplicates (and has drifted from)
`content/modules/m1.json` — a known issue noted in `CLAUDE.md`. This design
also resolves that, since per-locale content makes inlining untenable anyway.

## Decisions

### 1. Content file layout — per-locale module files

```
content/modules/
  ru/
    m0.json
    m1.json
    ... m2.json .. m10.json (stubs)
  en/
    m0.json
    m1.json
    ... m2.json .. m10.json (stubs)
```

- Same JSON schema as today (per `CONTENT_GUIDE.md`) — no schema changes.
- `m0-l1` and `m1-l1` are translated into `en/` as part of this work (full
  translation: motivation, theory, examples, exercises, mastery check, and
  **code comments inside example/exercise `code` fields**, since those are
  currently Russian).
- Stub modules/lessons (`"stub": true`) exist in both locales with translated
  `title` only, so the module map (ids, ordering, lesson ids) is identical
  between locales. This identity is what makes the parity check (decision 5)
  meaningful.
- `CONTENT_GUIDE.md` is updated to document the per-locale directory layout
  and the requirement that a lesson isn't "done" until both locale files
  exist and pass the parity check.

### 2. Engine content loading — fetch per locale, drop inline `COURSE_DATA`

- Remove the inlined `COURSE_DATA` object from `cpp26-engine.jsx`.
- On startup (and whenever the locale changes), the engine `fetch()`s
  `content/modules/<locale>/m{0..10}.json` and assembles the in-memory course
  model from the responses. A module file that 404s or is absent is treated
  as an empty stub (defensive, but in practice both locale trees will have all
  10 files per decision 1).
- `courseTitle` (and any other top-level course metadata currently inside
  `COURSE_DATA`) moves into the UI string catalog (decision 3) or a small
  per-locale `course.json`, whichever is simpler to wire — implementation
  plan decides; not a meaningful design fork.
- This is a pure refactor of the loading mechanism; rendering logic for
  lessons/exercises/etc. is unchanged.

### 3. UI string catalog

- A small bilingual dictionary, e.g. `UI_STRINGS = { ru: {...}, en: {...} }`,
  covering the fixed set of chrome strings identified in
  `cpp26-engine.jsx` (statuses: "Выполнено"/"Пройдено с пропусками"/"В
  процессе"/"Не начато"; verdicts: "Верно"/"Правильный ответ: ..."; buttons:
  "Запустить"/"Сверить"/"Компилирую…"; result headers; default learner name
  "Ученик"; etc.).
- A `t(key, ...args)` helper reads from the active locale with Russian as
  fallback for any missing key.
- Lives inline in `cpp26-engine.jsx` (consistent with "no build tooling"
  constraint) unless the implementation plan finds a cleaner spot.

### 4. Locale switcher + persistence

- A RU|EN toggle in the header, near the brand/profile area.
- Persisted in `localStorage` under a new key (e.g. `cpp26_locale`) —
  instant on load, works offline, mirrors how progress is persisted today.
- If a Supabase session exists, the locale preference is synced to the
  user's profile the same way progress is pushed (mirroring the existing
  `pushIfChanged` pattern), so it follows the user across devices. Local
  storage remains the source of truth for instant rendering; the Supabase
  value is read on login and reconciled (last-write-wins is acceptable —
  same as progress sync today).
- Default locale when nothing is stored: `ru` (matches current `<html
  lang="ru">` and the primary audience).
- `<html lang="...">` is updated to match the active locale.

### 5. Cross-locale parity check

- A small Node script (sibling to `prototype/run-on-godbolt.js`) that walks
  `content/modules/ru/*.json` and `content/modules/en/*.json` and verifies
  structural parity per module/lesson:
  - same set of module ids and lesson ids, same order
  - for non-stub lessons: same number of `examples`, same number of
    `exercises` with matching `id`s and `type`s, same number of `options` per
    `choice` exercise, same number of `masteryCheck.questions` with matching
    `answerIndex`/`answerLine`/`answer` where applicable (i.e. everything that
    isn't free-text prose must match exactly; prose fields are not compared).
- Added to the Definition-of-Done checklist in `CONTENT_GUIDE.md`: a lesson
  is not done until both locale files exist and the parity check passes.

### 6. Testing

- Extend the existing Playwright e2e suite (`e2e/`) with a test that:
  1. switches locale via the new toggle,
  2. asserts a UI chrome string changes (e.g. a status label),
  3. asserts lesson content changes (e.g. the lesson title),
  4. reloads the page and asserts the locale choice persisted.

## Out of scope

- Translating modules 2-10 content (they're stubs; only `m0-l1`/`m1-l1` exist
  today and are translated as part of this work).
- Any language beyond ru/en.
- Browser-language auto-detection for the default locale (default is always
  `ru` when nothing stored).
- Changing the lesson JSON schema itself (per-locale directories only).
- Supabase schema migration details/SQL — covered at the implementation-plan
  level alongside decision 4.
