# МАСТЕР-ПЛАН
## Обучающее приложение «C++26: от нуля до полного понимания»

**Версия:** v1 (черновик для итерации) · **Дата:** 1 июня 2026
**Контекст:** C++26 официально утверждён ISO 28 марта 2026 (редакция N5046). Набор фич зафиксирован — мы документируем готовый стандарт, а не движущуюся мишень.

**Два главных требования заказчика:**
1. Приложение — это **преподаватель**, пользователь — ученик. После прохождения ученик понимает каждую тему.
2. Покрытие C++26 **исчерпывающее**: каждая фича объясняется с примерами, мотивацией и упражнениями.

**Открытый вопрос (решаем вместе):** на чём писать само приложение. Раздел 3 даёт варианты и критерии — но это НЕ блокирует работу над контентом.

# ЧАСТЬ A. ПЛАН ОБУЧЕНИЯ (учебный маршрут)

## Принципы маршрута

- **Порядок по зависимостям**, не по алфавиту: сначала фундамент (compile-time, ergonomics), потом тяжёлые флагманы (рефлексия, std::execution).
- **Каждый модуль самодостаточен** как урок: мотивация → синтаксис → базовый пример → продвинутый пример → упражнения (basic + advanced) → проверка усвоения.
- **«Исчерпывающе» = ничего не замолчать.** Даже мелкие фичи и удалённые возможности получают карточку.
- **Mastery-gating:** модуль закрыт только после прохождения проверки усвоения (см. часть B). Это и есть механизм гарантии «после прохождения понимаю всё».

## Базовая планка ученика и фоновые врезки

**Планка входа: уверенный C++14 + знание C++17.** Это считаем известным и НЕ объясняем: `auto`, лямбды, move-семантика, вариативные шаблоны, базовый `constexpr`, `if constexpr`, fold-выражения, структурные привязки, `std::optional/variant/any`, `string_view`, CTAD.

Фоновые врезки покрывают **строго промежуток C++17 → C++26** — только то, что напрямую нужно для понимания конкретной C++26-фичи (load-bearing), не глубже. Ниже планки — не пишем сами, даём внешнюю ссылку.

Стандартный набор предпосылок (подтягивается по тегам урока):

### Модуль 0 — Контекст C++26 (вводный, без настройки)
> ⚠️ Без тем установки/настройки: ученик работает не на реальной машине, локально ничего не собирает.
- Что такое стандарт C++, краткая история, статус C++26 (утверждён ISO 28.03.2026, редакция N5046).
- Чем C++26 значим: три флагмана (рефлексия, контракты, std::execution) + сквозная тема безопасности.
- Как устроен курс: карта модулей, из чего состоит урок, что такое «зона повторения».
- Справочно (не задание): рефлексии нужны свежие компиляторы — но в приложении код не собирается, всё работает на готовых примерах (см. B6).
- **Проверка:** короткий вводный квиз на ориентацию (например, «что из перечисленного — флагман C++26»); никаких практических заданий по окружению.

### Модуль 1 — Эргономика ядра языка
- **Pack indexing** — прямая индексация пакета: `values...[0]`, `T...[N]`.
- **Placeholder `_`** — имя «мне всё равно», можно переиспользовать многократно.
- **`= delete("reason")`** — удалённые функции с поясняющим сообщением.
- **Структурные привязки**: как pack (P1061), атрибуты на привязках.
- **`#embed`** — встраивание бинарных/текстовых ресурсов на этапе препроцессора.
- Прочие мелочи: вариативные `friend`, улучшения шаблонов, сообщения в `static_assert`.
- *Пример (pack indexing):*
  ```cpp
  template <typename... T>
  constexpr auto first(T... v) { return v...[0]; }
  static_assert(first(10, 20, 30) == 10);
  ```

### Модуль 2 — Безопасность ядра
- **Erroneous behavior (P2795):** чтение неинициализированной переменной больше не UB, а «ошибочное поведение» — диагностируемый баг с определённым (но неправильным) результатом.
- **Запрет привязки возвращаемой ссылки к временному объекту** — ловит классический dangling.
- Другие мелкие safety-правки.
- Зачем: безопасность — сквозная тема C++26, это фундамент для понимания, *почему* стандарт двигается в эту сторону.

### Модуль 5 — Static Reflection *(флагман №1)*
Самый объёмный модуль, дробится на под-уроки:
- Оператор `^^` → значение типа `std::meta::info` (заголовок `<meta>`).
- Сплайсинг `[: r :]` — вставка отражённой сущности обратно в код (как тип/значение/имя).
- Запросы: члены типа, имена, типы, перечисление элементов enum.
- **Генерация кода** на основе интроспекции.
- Аннотации для рефлексии (поведение отдельно от обычных атрибутов).
- *Канонические примеры-упражнения:* `enum → string`, авто-сериализация struct в JSON, генерация компараторов.
- *Пример (форма):*
  ```cpp
  constexpr auto r = ^^int;   // рефлексия типа
  using T = [: r :];          // сплайс обратно в тип  → T == int
  ```
- ⚠️ Точные имена функций `std::meta::*` сверяем по cppreference при написании контента.

# ЧАСТЬ B. ПЛАН РЕАЛИЗАЦИИ ПРИЛОЖЕНИЯ

## B1. Главный принцип: «приложение — преподаватель»

Не чеклист с галочками, а среда, где **контент живёт внутри** и ведёт ученика. Минимальная единица — не «фича», а **урок**, который доводит до понимания и проверяет его.

## B2. Модель контента (иерархия данных)

```
Курс
 └── Модуль (0..10)
       └── Урок (1..N)
             ├── Фон / предпосылки (опц., сворачиваемый — C++20/23 врезки)
             ├── Мотивация (какую боль решает фича)
             ├── Теория (исчерпывающее объяснение)
             ├── Примеры (базовый + продвинутый, с подсветкой)
             ├── Упражнения (basic + advanced)
             └── Проверка усвоения (mastery check)
```

## B3. Анатомия урока (единый шаблон)

Каждый урок обязан содержать пять основных блоков (мотивация → теория → примеры → упражнения → проверка). Урок без продвинутого примера или без проверки считается недоделанным.

Шестой блок — **«Фон / предпосылки»** — опциональный, появляется только если у урока есть теги предпосылок (C++20/23). По умолчанию **свёрнут**: кто уверен — пропускает в один тап (связано с механикой пропуска, B5); кто не знает — разворачивает. Объём врезки строго load-bearing, по правилу из раздела «Базовая планка».

## B4. Система упражнений и обратной связи

**Принцип:** код компилирует НЕ ученик и не приложение — а Claude, один раз, при подготовке контента, напрямую через публичный API Compiler Explorer (godbolt.org), без локального тулчейна и без участия человека (см. скилл `compiling-cpp26-examples`, скрипт `prototype/run-on-godbolt.js`). Ожидаемые выводы и эталоны сохраняются проверенными. Ученик ничего не компилирует.

**Автопроверяемые задачи (часть прохождения, без компиляции кода ученика):**
1. **Предскажи вывод** — ученик вводит результат, сверка со заранее посчитанной строкой.
2. **Найди ошибку** — выбор строки/варианта с дефектом (например, dangling-ссылка из модуля 2).
3. **Выбор варианта / «что здесь UB или ill-formed»** — концептуальная проверка против известного ответа.

Лучший формат примера — **«предскажи → раскрой»**: ученик сначала предсказывает вывод, потом видит ответ с разбором. Это одновременно и дополнительный пример, и активная проверка (активное припоминание — самый надёжный режим усвоения).

**Опциональный челлендж (НЕ автопроверяется, НЕ влияет на прохождение):**
4. **Напиши сам** — свободное задание с эталонным решением и пояснением + кнопка «Открыть в Compiler Explorer». Свободный код нельзя надёжно проверить без компилятора, поэтому он вынесен из обязательного пути: кто хочет — пишет и проверяет себя сам.

Челлендж теперь не только статичен (эталонное решение + ссылка на Compiler Explorer): ученик может скомпилировать и исполнить свой вариант прямо в приложении («Запустить») и сверить вывод с проверенным эталоном («Сверить») — см. `docs/superpowers/specs/2026-06-07-runnable-challenge-design.md`. Это остаётся опциональной обратной связью и не влияет на mastery-gating и прохождение модуля — только автопроверяемые задачи (`predict-output`/`find-bug`/`choice`) считаются для прогресса.

Обратная связь — не «верно/неверно», а **разбор почему**: при ошибке показываем правильный ответ + объяснение + ссылку на нужный фрагмент теории.

## B5. Mastery-gating + пропуск задач (как гарантируется «понимаю всё», но без принуждения)

Баланс между свободой (пропуск) и целью (полное понимание):

- **Пропуск разрешён** на уровне отдельной задачи/урока: кнопка «Пропустить».
- Пропущенное получает **отдельный статус** `Пропущено` — он НЕ равен `Выполнено`. Никакого тихого «зачёта».
- Пропущенные задачи автоматически копятся в **«зоне повторения»**, к ним можно вернуться в любой момент.
- Модуль можно пометить пройденным с пропусками, но он показывает явный маркер: «N задач пропущено» — пробелы всегда видны.
- Опциональный «строгий режим»: модуль закрывается только при прохождении проверки (порог, например, 80%) без пропусков — для тех, кто хочет жёсткую гарантию.
- Прогресс различает три состояния: `Выполнено` / `Пропущено` / `Не начато`, и считается по усвоению, а не по «прокрутил страницу».
- Проверка усвоения строится **только на автопроверяемых задачах** (B4, типы 1–3). Челлендж «напиши сам» (тип 4) на прохождение не влияет — он добровольный.

## B6. Работа с кодом — приложение самодостаточно, без сборки

Ключевое ограничение: ученик **не на реальной машине** и ничего локально не компилирует. Поэтому приложение полностью автономно:
- Каждый пример хранит **готовый ожидаемый вывод**, проверенный заранее (один раз, на этапе подготовки контента).
- Упражнения «предскажи вывод» / «найди ошибку» проверяются по **сохранённому эталону**, а не компиляцией.
- Кнопка **«Открыть в Compiler Explorer»** — чисто **опциональная**: ссылка на godbolt для тех, кто *захочет* запустить вживую. Это не требование и не часть прохождения.
- Никаких флагов компилятора, установки или конфигурации в самом учебном потоке.

## B7. Прогресс и персистентность

- Состояние: статус модулей/уроков, результаты проверок, «зона повторения», заметки ученика.
- Хранение: `localStorage`/`IndexedDB` (см. B9), офлайн-кэш через service worker; опционально экспорт/импорт прогресса в файл.

## B8. Этапы реализации (milestones)

1. **M1 — Каркас:** модель данных + навигация по модулям + прогресс на 1 заглушечном модуле.
2. **M2 — Контент-пайплайн:** формат хранения уроков; наполняем модули 0–2 целиком.
3. **M3 — Упражнения и обратная связь:** 3 автопроверяемых типа (B4) + опциональный челлендж + разбор ошибок.
4. **M4 — Mastery-gating:** проверки усвоения, пороги, зона повторения.
5. **M5 — Тяжёлый контент:** модули 4, 5, 6 (контракты, рефлексия, execution).
6. **M6 — Полнота и полировка:** оставшиеся модули, интеграция с Compiler Explorer, финальная вычитка на «исчерпывающесть».

## B9. РЕШЕНО — стек: веб-приложение (PWA)

**Выбор зафиксирован.** Решающие факты: ученик не на реальной машине, нужна работа на телефоне И десктопе/браузере, разработчик идёт по Android с нуля. Это вычеркнуло нативный C++ (нужна сборка) и нативный Android (только телефон + новый стек Kotlin с нуля + нельзя показать в чате). Остаётся веб, упакованный как **PWA**.

Почему PWA закрывает всё:
- **Одна кодовая база → две площадки:** браузер на десктопе + установка на главный экран Android (офлайн, полноэкранный режим, ощущение приложения).
- **Самодостаточно:** прогресс в `localStorage`/`IndexedDB`, офлайн-кэш через service worker, ничего не компилируется.
- **Быстрый цикл:** прототип собираю и показываю прямо в чате.
- **Низкий порог:** оболочку пишу я; можно без сборочных инструментов (чистый HTML/JS или React через CDN). Единственная «настройка» — выложить статические файлы на бесплатный хостинг (GitHub Pages / Netlify) для установки на телефон.

Технологии: React + подсветка кода (Prism.js/highlight.js), контент как данные (JSON), `manifest.json` + service worker для установки и офлайна.

Путь реализации:
1. **Прототип** — React-артефакт в чате: отладка формата урока, навигации, прогресса, пропуска.
2. **Боевая версия** — автономный статический PWA: тот же UI, контент в JSON, манифест + service worker. Контент-данные переезжают без изменений.
